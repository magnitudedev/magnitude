# `@magnitudedev/harness` — Specification

## What It Is

A generic framework for executing a single LLM turn with tools. It drives a model, dispatches tool calls, tracks state via reducers, handles interrupts, and produces a canonical turn representation using `@magnitudedev/ai` types. Composable via hooks. No opinions on looping, prompt assembly, or context management.

The harness is fundamentally an **event processor**. Events flow through reducers that maintain state on refs. There are two modes of use — live execution (`runTurn` → `LiveTurn`) and replay (`createReplayTurn` → `ReplayTurn`) — backed by the same reducers and the same refs. The only difference is where events come from.

This design **enables** event-sourced architectures without forcing them. The harness produces events and exports reducers. Consumers who want event sourcing persist events and replay through reducers. Consumers who don't can ignore both.

**Dependencies:** `@magnitudedev/ai`, `effect`, `@effect/schema`
**Replaces:** `@magnitudedev/turn-engine`, parts of `@magnitudedev/tools`, parts of `packages/agent/src/execution`

## API Surface

### Harness

```typescript
function createHarness<TToolkit extends Toolkit, RHooks = never>(
  config: HarnessConfig<TToolkit, RHooks>
): Harness
```

```typescript
interface HarnessConfig<TToolkit extends Toolkit = Toolkit, RHooks = never> {
  readonly model: BoundModel
  readonly toolkit: TToolkit
  readonly hooks?: HarnessHooks<RHooks>
  readonly layer?: Layer.Layer<ToolkitRequirements<TToolkit> | RHooks>
  readonly initialState?: EngineState
  readonly messageDestination?: string
  readonly thoughtKind?: string
}
```

```typescript
interface Harness {
  readonly runTurn: (prompt: Prompt) => Effect.Effect<LiveTurn, ConnectionError | StreamError>
  readonly createReplayTurn: () => ReplayTurn
  readonly getToolDefinitions: () => readonly ToolDefinition[]
}
```

`layer` is compile-time enforced: when any tool's `execute` or any hook declares Effect service requirements (R channel), the compiler requires a `layer` that provides all of them. If no tools or hooks have requirements, `layer` is optional.

### LiveTurn

Returned by `runTurn`. Events come from the model stream and tool execution. Consumer observes events (to persist, forward to UI, etc.) and reads refs for live state.

```typescript
interface LiveTurn {
  readonly events: Stream.Stream<HarnessEvent>
  readonly canonicalTurn: Ref<CanonicalTurnState>
  readonly engineState: Ref<EngineState>
  readonly toolHandles: Ref<ToolHandleState>
}
```

Events are generated by the harness, fed through internal reducers (updating refs), and emitted on the stream. The consumer does not control event flow — it observes.

### ReplayTurn

Returned by `createReplayTurn`. Consumer feeds events (from storage, synthetic, etc.) and reads refs for reconstructed state.

```typescript
interface ReplayTurn {
  readonly feed: (event: HarnessEvent) => void
  readonly canonicalTurn: Ref<CanonicalTurnState>
  readonly engineState: Ref<EngineState>
  readonly toolHandles: Ref<ToolHandleState>
}
```

Same refs, same reducers as LiveTurn. The only difference is the consumer drives event flow.

### Why the split

`LiveTurn` and `ReplayTurn` are backed by the same reducers and the same ref types. The split exists because the event source is structurally different:

- **LiveTurn**: events come from the model stream. Consumer observes.
- **ReplayTurn**: events come from the consumer. Consumer drives.

Combining them into one type would require either runtime checks to prevent misuse (calling `run` after `feed`, calling `run` twice) or an API that looks like you can mix modes when you can't. The split makes intent explicit at creation time.

## Reducers

Reducers are pure `(state, event) => state` functions. The harness runs them internally on both `LiveTurn` and `ReplayTurn`. They are also exported from the package so consumers can fold events independently.

```typescript
interface Reducer<TState> {
  readonly initial: TState
  readonly step: (state: TState, event: HarnessEvent) => TState
}
```

### Built-in Reducers

```typescript
const CanonicalTurnReducer: Reducer<CanonicalTurnState>
const EngineStateReducer: Reducer<EngineState>
function createToolHandleReducer(toolkit: Toolkit): Reducer<ToolHandleState>
```

### CanonicalTurnState

The canonical turn is the best-available representation of what happened during the turn. Always readable — mid-stream, after interrupt, after crash recovery.

```typescript
interface CanonicalTurnState {
  readonly assistantMessage: AssistantMessage
  readonly toolResults: readonly ToolResultMessage[]
  readonly outcome: TurnOutcome | null  // null = turn still in progress
  readonly usage: ResponseUsage | null
}
```

Internally the reducer accumulates:
- Reasoning text from `ThoughtChunk`
- Message text from `MessageChunk`
- Per-tool-call field chunk text from `ToolInputFieldChunk`
- Active message state from `MessageStart`/`MessageChunk`/`MessageEnd`

On `ToolInputReady`: replaces partial input with validated, decoded input.
On `ToolExecutionEnded`: adds a `ToolResultMessage`.
On `TurnEnd(Interrupted)`: assembles any partial tool inputs from accumulated field chunks into best-effort input objects in the `AssistantMessage.toolCalls`.

### EngineState

Tracks tool call outcomes for resume/caching.

```typescript
interface EngineState {
  readonly toolCallMap: ReadonlyArray<readonly [string, string]>       // toolCallId → toolName
  readonly toolOutcomes: ReadonlyArray<readonly [string, ToolOutcome]> // toolCallId → outcome
  readonly deadToolCalls: readonly string[]
  readonly stopped: boolean
}

type ToolOutcome =
  | { readonly _tag: 'Completed'; readonly result: ToolResult }
  | { readonly _tag: 'DecodeFailure' }
```

On resume, `createHarness({ initialState: engineState })` uses cached outcomes to skip re-execution.

### ToolHandleState

Per-tool-call state model reduction.

```typescript
interface ToolHandleState {
  readonly handles: ReadonlyMap<string, ToolHandle>  // toolCallId → ToolHandle
}
```

`createToolHandleReducer(toolkit)` is a factory because handle creation depends on which toolkit entry's state model to use.

## Requirements Derivation

```typescript
type ToolRequirements<T> =
  T extends { execute: (...args: any[]) => Effect.Effect<any, any, infer R> } ? R : never

type ToolkitRequirements<T extends Toolkit> = {
  [K in ToolkitKeys<T>]: ToolRequirements<T['entries'][K]['tool']>
}[ToolkitKeys<T>]
```

This works because `defineToolkit` uses `const T` generics, preserving concrete tool types including their R channel. `ToolkitEntry` is generic over `TTool` (not constrained to the erased form), so R flows through.

## Consumer Use Cases

### 1. Live turn execution

```typescript
const harness = createHarness({ model, toolkit, hooks, layer })
const turn = yield* harness.runTurn(prompt)

yield* Stream.runForEach(turn.events, (event) =>
  Effect.gen(function* () {
    yield* eventSink.append(event)
    yield* displayProjection.update(event)
    // turn.canonicalTurn is live — UI can read mid-stream
    // turn.toolHandles is live — UI can show tool state as it changes
  })
)

// Stream completed — turn is done
const canonical = Ref.get(turn.canonicalTurn)  // final canonical turn
const engine = Ref.get(turn.engineState)       // tool outcomes for caching
```

### 2. Session resume — replay stored events

```typescript
const harness = createHarness({ model, toolkit, hooks, layer })

// Replay turn 1
const t1 = harness.createReplayTurn()
for (const e of storedTurn1Events) t1.feed(e)
const canonical1 = Ref.get(t1.canonicalTurn)  // complete

// Replay turn 2
const t2 = harness.createReplayTurn()
for (const e of storedTurn2Events) t2.feed(e)
const canonical2 = Ref.get(t2.canonicalTurn)  // complete

// Agent has canonical1 + canonical2 to build prompt context
```

### 3. Crash recovery — replay incomplete turn, feed synthetic close

```typescript
const harness = createHarness({ model, toolkit, hooks, layer })

const t = harness.createReplayTurn()
for (const e of storedEvents) t.feed(e)

// Detect incomplete turn
const state = Ref.get(t.canonicalTurn)
if (state.outcome === null) {
  t.feed({ _tag: 'TurnEnd', outcome: { _tag: 'Interrupted' }, usage: null })
}

// Now finalized — partial tool inputs assembled from accumulated chunks
const canonical = Ref.get(t.canonicalTurn)
// canonical.assistantMessage has tool calls with best-effort partial input
// canonical.toolResults has results only for tools that completed

// Persist synthetic close so next load doesn't repeat recovery
yield* eventSink.append({ _tag: 'TurnEnd', outcome: { _tag: 'Interrupted' }, usage: null })
```

### 4. Resume execution after crash

```typescript
// Step 1: Replay crashed turn to extract engine state
const crashed = harness.createReplayTurn()
for (const e of storedEvents) crashed.feed(e)
crashed.feed({ _tag: 'TurnEnd', outcome: { _tag: 'Interrupted' }, usage: null })

const previousEngine = Ref.get(crashed.engineState)
const previousCanonical = Ref.get(crashed.canonicalTurn)

// Step 2: Build prompt including interrupted turn's canonical representation
const prompt = buildPrompt([...priorTurns, previousCanonical])

// Step 3: Fresh live turn with cached engine state
const resumeHarness = createHarness({
  model, toolkit, hooks, layer,
  initialState: previousEngine  // completed tools won't re-execute
})
const turn = yield* resumeHarness.runTurn(prompt)
yield* Stream.runForEach(turn.events, persist)
```

### 5. UI reads tool handle state during live streaming

```typescript
const turn = yield* harness.runTurn(prompt)

yield* Effect.all([
  Stream.runForEach(turn.events, persist),
  Effect.gen(function* () {
    // Reactive read of tool handles for UI
    const handles = Ref.get(turn.toolHandles)
    for (const [id, handle] of handles.handles) {
      renderToolCard(handle.toolKey, handle.state)
      // state.phase: 'streaming' → 'executing' → 'completed'
    }
  })
], { concurrency: 'unbounded' })
```

## HarnessTool

Wraps ai's `ToolDefinition` with execution. Never-switched: bare `HarnessTool` is erased, `HarnessTool<TInput, TOutput, ...>` is concrete. Erased form is internal — not a public type name.

```typescript
type HarnessTool<TInput = never, TOutput = never, TEmission = never, TInputEncoded = TInput, TOutputEncoded = TOutput, E = never, R = never> =
  [TInput] extends [never]
    ? {
        readonly definition: ToolDefinition
        readonly execute: (input: unknown, ctx: ToolContext<never>) => Effect.Effect<unknown, unknown, unknown>
        readonly stream?: StreamHook<unknown, unknown, unknown, unknown, unknown>
        readonly emissionSchema?: Schema.Schema<unknown, unknown, never>
        readonly errorSchema?: Schema.Schema<unknown, unknown, never>
      }
    : {
        readonly definition: ToolDefinition<TInput, TOutput, TInputEncoded, TOutputEncoded>
        readonly execute: (input: TInput, ctx: ToolContext<TEmission>) => Effect.Effect<TOutput, E, R>
        readonly stream?: StreamHook<TInput, TEmission, unknown, E, R>
        readonly emissionSchema?: [TEmission] extends [never] ? undefined : Schema.Schema<TEmission, unknown, never>
        readonly errorSchema?: [E] extends [never] ? undefined : Schema.Schema<E, unknown, never>
      }

function defineHarnessTool<TInput, TOutput, TEmission = never, TInputEncoded = TInput, TOutputEncoded = TOutput, E = never, R = never>(
  config: { ... }
): HarnessTool<TInput, TOutput, TEmission, TInputEncoded, TOutputEncoded, E, R>
```

### ToolContext

```typescript
interface ToolContext<TEmission = never> {
  readonly emit: [TEmission] extends [never]
    ? never
    : (emission: TEmission) => Effect.Effect<void>
}
```

### StreamHook

Fires as the model streams tool call arguments. Allows tools to react to partial input before execution.

```typescript
interface StreamHook<TInput, TEmission, TStreamState, E = never, R = never> {
  readonly initial: TStreamState
  readonly onInput: (
    input: StreamingPartial<TInput>,
    state: TStreamState,
    ctx: ToolContext<TEmission>
  ) => Effect.Effect<TStreamState, E, R>
}
```

## Toolkit

Typed, composable collection of tool entries. Keys are `toolKey` — a compile-time literal discriminator distinct from `tool.definition.name` (the model-facing name).

`ToolkitEntry` is generic over `TTool` (not constrained to the erased form) so that `defineToolkit` with `const T` preserves the concrete tool types including their R channel.

```typescript
interface ToolkitEntry<
  TTool = HarnessTool,
  TStateModel extends StateModel | undefined = StateModel | undefined,
> {
  readonly tool: TTool
  readonly state?: TStateModel
  readonly group?: string
}

interface Toolkit<T extends Record<string, ToolkitEntry> = Record<string, ToolkitEntry>> {
  readonly entries: T
  readonly keys: readonly (keyof T & string)[]
  pick<K extends keyof T & string>(...keys: K[]): Toolkit<Pick<T, K>>
  omit<K extends keyof T & string>(...keys: K[]): Toolkit<Omit<T, K>>
}

function defineToolkit<const T extends Record<string, ToolkitEntry>>(entries: T): Toolkit<T>

function mergeToolkits<A extends Record<string, ToolkitEntry>, B extends Record<string, ToolkitEntry>>(
  a: Toolkit<A>, b: Toolkit<B>
): Toolkit<A & B>
```

### Extraction helpers

```typescript
type ToolkitKeys<T extends Toolkit> = keyof T['entries'] & string
type ToolkitTool<T extends Toolkit, K extends ToolkitKeys<T>> = T['entries'][K]['tool']
type ToolkitState<T extends Toolkit, K extends ToolkitKeys<T>> = T['entries'][K] extends { state: infer S } ? S : undefined

type ToolRequirements<T> =
  T extends { execute: (...args: any[]) => Effect.Effect<any, any, infer R> } ? R : never
type ToolkitRequirements<T extends Toolkit> = {
  [K in ToolkitKeys<T>]: ToolRequirements<T['entries'][K]['tool']>
}[ToolkitKeys<T>]
```

## StateModel

Per-tool state reducer. Curried `defineStateModel(toolKey, tool)` preserves full type inference — reducer receives `ToolLifecycleEvent<TInput, TOutput, TEmission>` and state includes literal `toolKey`.

```typescript
type Phase = 'streaming' | 'executing' | 'completed' | 'error' | 'rejected' | 'interrupted'

interface BaseState {
  readonly phase: Phase
}

interface StateModel<TState extends BaseState = BaseState, TInput = unknown, TOutput = unknown, TEmission = unknown> {
  readonly initial: TState
  readonly reduce: (state: TState, event: ToolLifecycleEvent<TInput, TOutput, TEmission>) => TState
}

function defineStateModel<TToolKey extends string, TInput, TOutput, TEmission = never>(
  toolKey: TToolKey,
  tool: { definition: { inputSchema: { Type: TInput }; outputSchema: { Type: TOutput } }; emissionSchema?: { Type: TEmission } }
): <TExtra extends Record<string, unknown>>(config: {
  initial: TExtra
  reduce: (
    state: { readonly toolKey: TToolKey } & BaseState & TExtra,
    event: ToolLifecycleEvent<TInput, TOutput, TEmission>
  ) => { readonly toolKey: TToolKey } & BaseState & TExtra
}) => StateModel<{ readonly toolKey: TToolKey } & BaseState & TExtra, TInput, TOutput, TEmission>
```

## Hooks

Single interface. All optional. Hooks declare their own Effect requirements via the `R` type parameter. The harness provides the `layer` from config to all hook effects.

```typescript
interface ExecuteHookContext {
  readonly toolCallId: string
  readonly toolName: string
  readonly toolKey: string
  readonly group: string
  readonly input: unknown
}

type InterceptorDecision =
  | { readonly _tag: 'Proceed'; readonly modifiedInput?: unknown }
  | { readonly _tag: 'Reject'; readonly rejection: unknown }

interface HarnessHooks<R = never> {
  readonly beforeExecute?: (ctx: ExecuteHookContext) => Effect.Effect<InterceptorDecision, never, R>
  readonly afterExecute?: (ctx: ExecuteHookContext & { readonly result: ToolResult }) => Effect.Effect<void, never, R>
  readonly onEvent?: (event: HarnessEvent) => Effect.Effect<void, never, R>
  readonly onEmission?: (ctx: { readonly toolCallId: string; readonly toolName: string; readonly toolKey: string; readonly value: unknown }) => Effect.Effect<void, never, R>
  readonly formatResult?: (toolName: string, toolKey: string, result: ToolResult) => readonly ToolResultPart[]
  readonly onResult?: (ctx: { readonly toolCallId: string; readonly toolName: string; readonly toolKey: string; readonly result: ToolResult; readonly parts: readonly ToolResultPart[] }) => Effect.Effect<void, never, R>
}
```

## Model Errors

ai error types forwarded with full fidelity through the Effect error channel. No wrapping.

**Connection errors** (pre-stream): `AuthFailed`, `RateLimited`, `UsageLimitExceeded`, `ContextLimitExceeded`, `InvalidRequest`, `TransportError`

**Stream errors** (mid-stream): `TransportError`, `ParseError`

Consumer handles via `Effect.catchTag`:
```typescript
harness.runTurn(prompt).pipe(
  Effect.catchTag("RateLimited", (e) => /* e.retryAfterMs available */),
  Effect.catchTag("ContextLimitExceeded", (e) => /* compact and retry */),
)
```

Harness re-exports these types from ai.

## Execution Flow (runTurn)

1. Extract ai `ToolDefinition[]` from toolkit
2. Build `toolKey ↔ toolName` maps from toolkit
3. Call `model.stream(prompt, toolDefs)` — errors propagate as `ConnectionError | StreamError`
4. For each `ResponseStreamEvent`, translate to `HarnessEvent` and feed through internal reducers:
   - Reasoning chunks → `ThoughtStart/Chunk/End`
   - Message text → `MessageStart/Chunk/End`
   - Tool call start → `ToolInputStarted`
   - Tool call field delta → `ToolInputFieldChunk`
   - Tool call end → schema-validate, emit `ToolInputReady` or `ToolInputDecodeFailure`
5. On `ToolInputReady`:
   - Check `engineState` for cached outcome → emit `ToolExecutionStarted(cached: true)` + `ToolExecutionEnded`
   - Otherwise: `hooks.beforeExecute` → Reject? → `ToolExecutionEnded(Rejected)`
   - Otherwise: execute tool within `layer` → `ToolExecutionStarted` → emissions → `ToolExecutionEnded`
   - `hooks.afterExecute`
6. During streaming: fire `StreamHook.onInput` if tool has one
7. After execution: `hooks.formatResult` → `hooks.onResult` → build `ToolResultMessage`
8. On model stream complete → `TurnEnd(Completed)`
9. On interrupt → `TurnEnd(Interrupted)` — reducers assemble partial state
10. Every event is fed through reducers (updating refs) and emitted on the output stream

## Scenarios

### Scenario 1: Clean turn

```
runTurn(prompt) → LiveTurn.events:
  ThoughtStart → ThoughtChunks → ThoughtEnd
  ToolInputStarted("a") → ToolInputFieldChunks → ToolInputReady("a")
  ToolInputStarted("b") → ToolInputFieldChunks → ToolInputReady("b")
  ToolExecutionStarted("a") → ToolEmissions → ToolExecutionEnded("a", Success)
  ToolExecutionStarted("b") → ToolExecutionEnded("b", Success)
  TurnEnd(Completed)
```

`canonicalTurn`: full AssistantMessage (reasoning, 2 tool calls with validated input) + 2 ToolResultMessages. `outcome: Completed`.

### Scenario 2: Interrupt mid-tool-input streaming

Model streaming tool call "a" field chunks. Interrupt hits before `ToolInputReady`.

```
ToolInputStarted("a")
ToolInputFieldChunk("a", field: "path", delta: "/src/foo.ts")
ToolInputFieldChunk("a", field: "content", delta: "const x")
TurnEnd(Interrupted)
```

`canonicalTurn`: AssistantMessage with 1 tool call, input assembled from chunks: `{ path: "/src/foo.ts", content: "const x" }`. No ToolResultMessages. `outcome: Interrupted`.

### Scenario 3: Partial completion — 2 of 3 tools done, interrupt during 3rd

```
ToolInputStarted("a") → ... → ToolInputReady("a")
ToolInputStarted("b") → ... → ToolInputReady("b")
ToolInputStarted("c") → ... → ToolInputReady("c")
ToolExecutionStarted("a") → ToolExecutionEnded("a", Success)
ToolExecutionStarted("b") → ToolExecutionEnded("b", Success)
ToolExecutionStarted("c")
TurnEnd(Interrupted)
```

`canonicalTurn`: 3 tool calls (all full input), 2 ToolResultMessages. `outcome: Interrupted`.
`engineState`: outcomes for "a" and "b", none for "c".

### Scenario 4: Process crash — no TurnEnd persisted

```typescript
const t = harness.createReplayTurn()
for (const e of storedEvents) t.feed(e)

const state = Ref.get(t.canonicalTurn)
if (state.outcome === null) {
  t.feed({ _tag: 'TurnEnd', outcome: { _tag: 'Interrupted' }, usage: null })
}
// Finalized — partial tool inputs assembled from accumulated chunks
```

### Scenario 5: Tool input decode failure

```
ToolInputStarted("a") → ToolInputFieldChunks → ToolInputReady("a")
ToolInputStarted("b") → ToolInputFieldChunks
ToolInputDecodeFailure("b", detail: SchemaError)
ToolExecutionStarted("a") → ToolExecutionEnded("a", Success)
TurnEnd(ToolInputDecodeFailure)
```

`canonicalTurn`: 2 tool calls. "a" has full input + result. "b" has accumulated chunk input, no result.

### Scenario 6: Hook rejects tool execution

```
ToolInputStarted("a") → ... → ToolInputReady("a")
ToolInputStarted("b") → ... → ToolInputReady("b")
ToolExecutionStarted("a") → ToolExecutionEnded("a", Success)
ToolExecutionStarted("b") → ToolExecutionEnded("b", Rejected)
TurnEnd(GateRejected)
```

### Scenario 7: Replay multiple turns from stored session

```typescript
const t1 = harness.createReplayTurn()
for (const e of turn1Events) t1.feed(e)
// Ref.get(t1.canonicalTurn) — complete turn 1

const t2 = harness.createReplayTurn()
for (const e of turn2Events) t2.feed(e)
// Ref.get(t2.canonicalTurn) — complete turn 2

const t3 = harness.createReplayTurn()
for (const e of turn3Events) t3.feed(e)
// outcome === null — crashed turn
t3.feed({ _tag: 'TurnEnd', outcome: { _tag: 'Interrupted' }, usage: null })
```

### Scenario 8: Resume with cached tool outcomes

```typescript
// Replay crashed turn
const crashed = harness.createReplayTurn()
for (const e of storedEvents) crashed.feed(e)
crashed.feed({ _tag: 'TurnEnd', outcome: { _tag: 'Interrupted' }, usage: null })

// Start fresh live turn with cached engine state
const resumeHarness = createHarness({
  model, toolkit, hooks, layer,
  initialState: Ref.get(crashed.engineState)
})
const turn = yield* resumeHarness.runTurn(prompt)
// Completed tools get cached execution, incomplete tools re-execute
```

### Scenario 9: Text-only response

```
MessageStart → MessageChunks → MessageEnd → TurnEnd(Completed, toolCallsCount: 0)
```

### Scenario 10: Stream error mid-response

```
ThoughtStart → ThoughtChunks
— StreamError propagates through Effect error channel —
TurnEnd(Interrupted)
```

## Type Safety

### Never-switching
- `HarnessTool<TInput = never, ...>` — bare is erased, parameterized is concrete
- `StateModel<TState = BaseState, TInput = unknown, ...>` — bare is erased, parameterized is concrete

### Curried inference
- `defineStateModel('shell', shellTool)({ initial, reduce })` — reducer gets `ToolLifecycleEvent<ShellInput, ShellOutput, never>`, state includes `toolKey: 'shell'`

### Generic events
- `ToolInputFieldChunk<TInput>` — `field: string & keyof TInput`, `path: DeepPaths<TInput>`
- `ToolExecutionEnded<TOutput>` — `result: ToolResult<TOutput>`

### Compile-time R enforcement
- Each tool's `execute` declares its Effect requirements (R channel)
- Each hook declares its Effect requirements via `HarnessHooks<R>`
- `ToolkitRequirements<T>` extracts and unions all tool R channels from the toolkit
- `createHarness` requires `layer: Layer.Layer<ToolkitRequirements<TToolkit> | RHooks>`
- `ToolkitEntry` is generic over `TTool` (not constrained to erased form) so `const T` inference preserves R through `defineToolkit`
- Missing a service in the layer is a compile-time error

### Tool schema alignment
- `defineHarnessTool` enforces that `errorSchema` matches `execute`'s `E` channel and `emissionSchema` matches `ToolContext<TEmission>.emit` — both are the same generic parameter, so mismatches are compile-time errors
- Consumers must use `defineHarnessTool` (not manual object construction) to get this enforcement

### No `any` in public API
- Erased forms use `unknown`
- Schema types use `Schema.Schema<unknown, unknown, never>` not `Schema.Schema.Any`
- No `as any`, no `as unknown as X`

### Toolkit key preservation
- `defineToolkit<const T>(entries)` preserves literal keys
- `ToolkitKeys<T>` extracts exact key union
- `toolKey` is a compile-time discriminator throughout (events, hooks, handles, state models)
