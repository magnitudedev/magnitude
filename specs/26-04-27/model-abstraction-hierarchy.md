# Model Abstraction Hierarchy

This document defines the layers of abstraction for model invocation in Magnitude, the contracts between them, and how a single call flows through the stack.

## Layers

### AssistantAtom (NEW)

Semantic representation of agent intent. The unit of agent state.

```ts
type AssistantAtom = ThoughtAtom | MessageAtom | ToolCallAtom

interface ThoughtAtom {
  kind: 'thought'
  id: string
  name: string | null      // alignment / tasks / etc., or null for unnamed
  text: string
}

interface MessageAtom {
  kind: 'message'
  id: string
  destination: { kind: 'user' } | { kind: 'worker'; taskId: string } | { kind: 'parent' }
  text: string
}

interface ToolCallAtom {
  kind: 'tool_call'
  id: string               // toolCallId, generated by us
  toolName: string
  input: unknown
  query: string | null
}
```

Atoms are universal. The agent emits and consumes them regardless of which codec is in use.

### ModelFunction

Named, typed agent-level call. Examples: `CodingAgentChat`, `SimpleChat`, `ExtractMemoryDiff`, `PatchFile`.

Defines:
- Input type (e.g., `{ systemPrompt, history: AssistantAtom[][], inboxState, options }`)
- Output type (e.g., `Stream<AssistantAtom>` or `{ text: string; usage }`)
- Mode: `'stream'` or `'complete'`

A ModelFunction is paradigm-aware in the sense that it specifies what kind of structured output it needs. The CodingAgentChat function streams atoms; PatchFile completes to a string.

### BoundModel

Resolved invocation handle. Carries everything needed to execute a call:
- `ProviderModel` (id, capabilities, costs)
- `Model | null` (canonical, if open-source)
- `Driver` (the runner)
- `Codec` (the format)

The resolver picks Driver + Codec based on (ProviderModel, paradigm needs of the ModelFunction, configuration).

### Driver

Runs a model. Owns:
- Transport (HTTP, SSE, streaming)
- Auth (api-key, oauth, etc.)
- Connection lifecycle
- Retry / error classification
- Speaks **one specific wire shape** (one HTTP endpoint family with one request/response contract)

Driver does NOT know about atoms, prompts, tools, or paradigms. It accepts an encoded request body and emits a stream of parsed wire chunks. The codec encodes the request and decodes the chunks.

```ts
interface Driver<WireRequest, WireChunk> {
  send(req: WireRequest, opts: DriverCallOptions): Effect<Stream<WireChunk>>
}
```

A driver class is essentially **"talks to one kind of HTTP endpoint."** Different endpoint shape = different driver, even if served by the same provider.

Variants:
- `OpenAIChatCompletionsDriver` — `/v1/chat/completions` (OpenAI-compatible: `messages[]`, `tools[]`, structured `tool_calls`, `reasoning_content` deltas). Serves Fireworks, OpenRouter, Together, Groq, Cerebras, vLLM, Ollama, LM Studio, OpenAI itself — any "OpenAI-compatible endpoint". The driver class is named for the **wire shape**, not the company.
- `OpenAICompletionsDriver` *(future)* — `/v1/completions` (raw text: `prompt: string`, returns `text` deltas; no roles, no chat template, no structured tool fields). Different endpoint, different shape, different driver — even though Fireworks serves both.
- `AnthropicMessagesDriver` *(future)* — Anthropic's `/v1/messages` API (different wire shape from OpenAI chat completions).

**Currently shipping:** `OpenAIChatCompletionsDriver` only.

### Codec

Encodes atoms into a wire request body; decodes a wire response stream into atoms.

```ts
interface Codec {
  encode(input: { atoms: AssistantAtom[][]; inbox: InboxState; ... }): WireRequest
  decode(stream: Stream<WireChunk>): Stream<AssistantAtom>
}
```

A codec is paradigm-specific. There are three:

#### xml-act codec
- **Universal**: works with any driver that can carry text-in / text-out (chat completions content field, raw completions, anthropic messages content blocks).
- Encode: render atoms as a single text blob containing structured tags. Place text into assistant message.
- Decode: parse the text stream looking for those tags → emit atoms.
- Used today everywhere; being orphaned by the native paradigm refactor.

#### native codec
- **Pairs with chat-completions-shaped drivers** (uses `messages[]`, structured `tool_calls`, `reasoning_content`).
- Encode: emit chat-completions request with structured assistant turns and tool definitions.
- Decode: read `delta.content`, `delta.tool_calls[i]`, `delta.reasoning_content` from the response stream.
- Family-agnostic; the server's chat template handles per-family rendering.
- **Currently shipping** as `NativeChatCompletionsCodec`.

#### completions codec (per-family)
- **Pairs with raw-completions-shaped drivers** (`/v1/completions` style — `prompt: string` in, text out).
- Per-family: dispatches to a `ModelAdapter` for encoding tokens and parsing tool calls.
- Encode: render atoms as text in the model's native trained format (Kimi tool tokens, GLM tool tags, etc.).
- Decode: parse raw text output using the family's adapter.
- Used when bypassing server-side chat-template + tool parsing (grammar-constrained decoding, lower abstraction tax, custom token control).
- **Future work**, not in current refactor.

### ModelAdapter

Per-family runtime module for the **completions codec**.

```ts
interface ModelAdapter {
  family: string              // 'kimi', 'glm', 'qwen', ...

  // Render atoms as text in this family's native trained format
  encode(atoms: AssistantAtom[]): string

  // Parse raw text output (model's native format) into atoms
  decode(stream: Stream<string>): Stream<AssistantAtom>

  // (Optionally) chat template, for token estimation, prompt previews, etc.
  template?: string
}
```

One adapter per supported family: kimi, glm, qwen, minimax, deepseek, ...

Each `MODEL.md` documents what its adapter must do (trained tokens, tool call format, think format, etc.). Adapters live in `packages/model-adapters/families/<family>/`.

The xml-act and native codecs do NOT use adapters — they're family-agnostic.

### ProviderProtocol

The wire-level API shape the provider speaks. Existing concept. Three variants:
- `openai-generic` — OpenAI-compatible chat completions (Fireworks, Z.AI, Moonshot, OpenRouter, Vercel, Magnitude, local servers, etc.)
- `anthropic` — Anthropic Messages API (Anthropic, Kimi-for-Coding, MiniMax)
- `openai` — OpenAI's official APIs (chat completions OR responses, depending on auth)

Carries `capabilities`: typed functions that build protocol-specific request options for things like `grammar`, `tools`, `stop_sequences`, `max_tokens`, `reasoning_effort`.

The protocol is selected by the provider's `resolveProtocol(auth)` method.

### Model (canonical)

Provider-independent identity for an open-source model.

```ts
interface Model {
  id: ModelId
  name: string
  family: string             // e.g., 'kimi'
  hfRepo: string             // e.g., 'moonshotai/Kimi-K2.6'
  template: string           // chat template content
}
```

`Model.family` is the join key with `ModelAdapter.family`. Same Model can be served by multiple Providers.

### ProviderModel

A specific model offered by a specific provider — `(Provider × Model | null)`.

```ts
interface ProviderModel {
  id: string                       // provider-specific id
  providerId: string
  modelId: ModelId | null          // canonical Model id, if open-source
  contextWindow, costs, ...
  supportsToolCalls, supportsReasoning, supportsVision, supportsGrammar
}
```

### Provider

A service endpoint definition. Examples: `fireworks-ai`, `zai`, `openrouter`, `anthropic`, `openai`, `lmstudio`, `ollama`.

```ts
interface ProviderDefinition {
  id, name, defaultBaseUrl
  models: ProviderModel[]
  authMethods: AuthMethodDef[]
  resolveProtocol(auth): { bamlProvider, protocol: ProviderProtocol }
}
```

## Contracts (What Touches What)

### Provider → ProviderProtocol
`provider.resolveProtocol(auth)` produces a ProviderProtocol. Determines which wire shape the driver targets.

### Provider → ProviderModel[]
A provider lists the ProviderModels it serves.

### ProviderModel → Model
`providerModel.modelId` references `Model.id` for open-source models.

### Model → ModelAdapter
`model.family` keys into the adapter registry. The completions codec resolves this lookup.

### Driver → ProviderProtocol
A driver knows how to call one (or more) protocol shapes. E.g., `OpenAIChatDriver` speaks `openai-generic`.

### Driver → Codec
A driver does NOT contain a codec. The driver and codec are composed at call time. The codec encodes a request, the driver sends it, the driver returns a chunk stream, the codec decodes it.

### Codec → AssistantAtom
Codecs encode and decode atoms. Atoms are the universal currency.

### Completions Codec → ModelAdapter
The completions codec dispatches to the right adapter based on `model.family`.

### ModelFunction → BoundModel
A model function executes against a bound model.

### BoundModel → (ProviderModel, Model, Driver, Codec)
The bound model bundles the resolved driver and codec together with the model identity.

### AssistantAtom → MemoryProjection
Assistant turns are stored as `AssistantAtom[]` in memory (replacing today's text storage). Memory rendering for view-time delegates to the codec's encode step.

## The Three Paradigms

```
                ┌──────────────────┬──────────────────┬──────────────────┐
                │ xml-act          │ native           │ completions      │
                ├──────────────────┼──────────────────┼──────────────────┤
Codec           │ xml-act codec    │ native codec     │ completions codec│
Family-aware    │ no               │ no               │ yes (uses adapter)│
Adapters used   │ —                │ —                │ kimi/glm/qwen/...│
Server parses   │ —                │ tool_calls       │ —                │
   tool calls?  │   (text only)    │   reasoning_cnt  │   (raw text)     │
Compatible      │ chat-completions │ chat-completions │ completions      │
   drivers      │   anthropic-msgs │   anthropic-msgs │   (raw text)     │
                │   completions    │                  │                  │
Use case        │ universal,       │ best for trained │ grammar-          │
                │ today's default  │ open models via  │   constrained,    │
                │ (being orphaned) │ chat completions │ raw text control  │
                └──────────────────┴──────────────────┴──────────────────┘
```

**A paradigm is defined by the codec.** Drivers don't dictate paradigm directly, but they constrain which codecs can plug in:

- A **chat-completions driver** (`/v1/chat/completions`) carries enough wire shape (structured `tool_calls`, `reasoning_content` deltas) to support both **xml-act** (just stuff text into `content`) and **native** (use the structured fields).
- A **completions driver** (`/v1/completions`, raw text in/out) only carries text, so it works with **xml-act** (text-only) or **completions** (per-family text codec). It can't support **native** because there are no structured fields.
- An **anthropic-messages driver** has its own wire shape; both xml-act and a native-equivalent codec could pair with it (different impl from `NativeChatCompletionsCodec` since wire shape differs).

So the mix-and-match rule: **codec encodes/decodes a wire shape; driver carries that wire shape over HTTP.** Anywhere those two agree on shape, they compose. The number of (driver × codec) pairs is bounded by wire-shape compatibility, not arbitrary.

## Mix-and-Match Examples

**Same driver, different codec:**
- `OpenAIChatCompletionsDriver` + `xml-act codec` against Fireworks → today's xml-act on Fireworks. Wire = one user message containing tag-laden text.
- `OpenAIChatCompletionsDriver` + `native codec` against Fireworks → today's native paradigm on Fireworks (the work this refactor ships).
- Same driver, swap the codec, and the agent paradigm swaps. No transport changes.

**Same codec, different driver:**
- `native codec` + `OpenAIChatCompletionsDriver` against Fireworks endpoint URL.
- `native codec` + `OpenAIChatCompletionsDriver` against `localhost:11434/v1` (Ollama). No code change — just config (different endpoint + auth in `DriverCallOptions`).
- `native codec` + `AnthropicMessagesDriver` against Anthropic *(future)* — codec interface is the same, but you write a separate Anthropic-native codec impl since wire shape (Anthropic messages content blocks) differs from chat completions.

**Same provider, different driver class:**
- Fireworks serves both `/v1/chat/completions` and `/v1/completions`. They're different drivers (`OpenAIChatCompletionsDriver` and `OpenAICompletionsDriver`) because the wire shapes are different — even though the auth, host, and rate limiting are shared.
- So `BoundModel` for Fireworks-Kimi-K2.6 can pick `chat driver + native codec` for one workload and `completions driver + completions codec(KimiAdapter)` for another (e.g. grammar-constrained extraction).

**What's currently shipping:**
- Driver: `OpenAIChatCompletionsDriver` only.
- Codec: `NativeChatCompletionsCodec` only.
- Paradigm: `native` only.
- xml-act paradigm is being orphaned (the existing BAML-driven path stays running until removed in a later phase).
- `OpenAICompletionsDriver`, `completions codec`, `ModelAdapter`s, anthropic-messages driver — all future work, not in this refactor.

## How A Call Flows

```
1. Agent invokes ModelFunction
   - assembles input from MemoryProjection (AssistantAtom[][] + inbox)

2. Resolver picks BoundModel
   - given the (ProviderModel, paradigm needs, config), selects:
     • Driver (e.g., OpenAIChatDriver)
     • Codec (e.g., native codec)

3. Codec.encode(input) → WireRequest
   - native codec produces messages[], tools[], assistant tool_calls, etc.
   - completions codec dispatches to ModelAdapter.encode for the family

4. Driver.stream(WireRequest) → Stream<WireChunk>
   - HTTP, auth, retries, error classification
   - streams chunks back unmodified

5. Codec.decode(Stream<WireChunk>) → Stream<AssistantAtom>
   - native codec reads delta.content, delta.tool_calls, delta.reasoning_content
   - completions codec dispatches to ModelAdapter.decode for the family
   - xml-act codec parses magnitude tags from text

6. Agent observes Stream<AssistantAtom>
   - feeds atoms to projections
   - dispatches tool calls via existing tool-execution infrastructure
   - emits events for downstream projections
```

## Naming Crisp Summary

```
Provider          — service endpoint (Fireworks, Z.AI, ...)
ProviderProtocol  — wire-level API shape (openai-generic, anthropic, ...)
Model             — open-source model identity (kimi-k2.6)
ProviderModel     — (Provider × Model) — what's available where
ModelAdapter      — per-family module (parser/encoder for native trained format)
Driver            — runs the model: transport, auth, connection, retries
Codec             — encodes/decodes atoms ↔ wire (paradigm-specific)
ModelFunction     — typed agent-level call (CodingAgentChat, ...)
BoundModel        — bundle: ProviderModel × Model × Driver × Codec
AssistantAtom     — semantic intermediate: thought | message | tool_call
```

```
Paradigm          — choice of codec
                    xml-act     (universal text-tag codec)
                    native      (universal structured chat completions codec)
                    completions (per-family raw text codec, uses ModelAdapter)
```

## What Each Concept Does (Single Line Each)

- **Provider**: where to send requests
- **ProviderProtocol**: wire-level API shape and capabilities
- **Model**: the thing being run (canonical, open-source)
- **ProviderModel**: a Provider×Model with concrete pricing and availability
- **ModelAdapter**: per-family parser/encoder for that family's trained native format
- **Driver**: orchestration + transport + auth — the runner
- **Codec**: format encoder/decoder — atoms ↔ wire
- **ModelFunction**: typed agent-level call
- **BoundModel**: resolved bundle ready to invoke
- **AssistantAtom**: semantic unit of agent intent
- **Paradigm**: choice of codec
