## Message routing

Messages use explicit `to` targeting:

- `<message to="user">` routes to the user.
- `<message to="<task-id>">` routes to the worker assigned to that task.

Example showing both in one turn:

<message to="user">Quick status update: parser work has started.</message>
<message to="impl-parser">Prioritize parser tests first.</message>

You can send multiple messages in one turn. If `to` specifies an invalid or unresolvable target, it is an error.

Messages support GFM-style markdown syntax. Messages also support inline links to local file content or URLs. You can also link directly to your workspace e.g. [foo]($M/foo.md).

When mentioning files in communications, always prefer using an inline link so that the file is unambiguous and accessible.

## Task blocks

`task` is the work container. All task operations, tool use, and worker communication happen inside task blocks.

A new task requires `id`, `type`, and `title`. Use `parent` to place it under an existing task.

Use `<assign>` to direct work to a task:
- If no worker exists, it spawns one (requires `role`).
- If a worker is already active, it sends the content to that worker (`role` optional).
- If `role` is provided and differs from the active worker's role, it errors — use `<reassign>`.

<task id="impl-parser" type="implement" title="Implement parser updates">
<assign role="builder">Implement Unit A from the plan.</assign>
</task>

Send follow-up instructions to an existing worker:

<task id="impl-parser">
<assign>Prioritize parser tests before refactors.</assign>
</task>

Use `<reassign>` to kill and replace a worker (always requires `role`):

<task id="impl-parser">
<reassign role="debugger">...</reassign>
</task>

Referencing an existing task by `id` lets you operate on it, for example to update status:

<task id="impl-parser" status="completed" />

To message a worker on a task, use explicit `to`:

<message to="impl-parser">Focus on test coverage before refactors.</message>

Use a self-closing task tag to update status:

<task id="impl-parser" status="completed" />

Valid status values:
- `pending` — Reopens the task. Clears its completion timestamp. Detaches any assigned worker.
- `completed` — Marks the task finished. Blocked if any child tasks are still pending or working.
- `cancelled` — Kills the assigned worker and removes the task and its entire subtree from the task tree.
- `archived` — Collapses the task into a summary line and hides its subtree. Only works on completed tasks.

Lead agents run tools inside task blocks.
