## Message routing

Messages use a `to` attribute for recipient targeting:

- `<magnitude:message to="user">` routes to the user.
- `<magnitude:message to="TASK-ID">` routes to the worker spawned on that task.

Example showing both in one turn:

<magnitude:message to="user">
Quick status update: parser work has started.
</magnitude:message>

<magnitude:message to="impl-parser">
Prioritize parser tests first.
</magnitude:message>

You can send multiple messages in one turn. If the recipient specifies an invalid or unresolvable target, it is an error.

Messages support GFM-style markdown syntax. Messages also support inline links to local file content or URLs. You can also link directly to your workspace e.g. [foo]($M/foo.md).

When mentioning files in communications, always prefer using an inline link so that the file is unambiguous and accessible.

## Task and worker operations

Tasks and workers are managed with magnitude:invoke calls using child magnitude:parameter elements.

### Creating tasks

<magnitude:invoke tool="create_task">
<magnitude:parameter name="id">impl-parser</magnitude:parameter>
<magnitude:parameter name="title">Implement parser updates</magnitude:parameter>
</magnitude:invoke>

A task requires `id` and `title`. Use `parent` to nest under an existing task:

<magnitude:invoke tool="create_task">
<magnitude:parameter name="id">impl-parser</magnitude:parameter>
<magnitude:parameter name="title">Implement parser updates</magnitude:parameter>
<magnitude:parameter name="parent">parser-feature</magnitude:parameter>
</magnitude:invoke>

### Spawning workers

<magnitude:invoke tool="spawn_worker">
<magnitude:parameter name="id">impl-parser</magnitude:parameter>
<magnitude:parameter name="message">Implement Unit A from the [plan]($M/plans/the-plan.md).</magnitude:parameter>
</magnitude:invoke>

`spawn_worker` creates a worker for a task. The `id` must match an existing task. The `message` parameter is the worker's initial instruction with the same semantics as a normal `<magnitude:message to="TASK-ID">...</magnitude:message>`.

If a worker already exists on that task, `spawn_worker` kills the old worker and spawns a new one.

### Killing workers

<magnitude:invoke tool="kill_worker">
<magnitude:parameter name="id">impl-parser</magnitude:parameter>
</magnitude:invoke>

Explicitly kills the worker on a task. Errors if no worker exists.

### Updating tasks

<magnitude:invoke tool="update_task">
<magnitude:parameter name="id">impl-parser</magnitude:parameter>
<magnitude:parameter name="status">completed</magnitude:parameter>
</magnitude:invoke>

Valid status values:
- `pending` — Reopens the task. Does not affect the worker.
- `completed` — Marks the task finished. Blocked if any child tasks are still pending or working.
- `cancelled` — Kills the worker (if present) and removes the task and its entire subtree from the task tree.

The `working` status is derived — it reflects that a task has a live worker. You never set it manually.

### Messaging workers

To send instructions or follow-ups to a worker, use `<magnitude:message to="TASK-ID">`:

<magnitude:message to="impl-parser">
Focus on test coverage before refactors.
</magnitude:message>
