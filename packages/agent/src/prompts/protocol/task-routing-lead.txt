## Message routing

Messages use colon-delimited recipient targeting:

- `<|message:user>` routes to the user.
- `<|message:TASK-ID>` routes to the worker spawned on that task.

Example showing both in one turn:

<|message:user>
Quick status update: parser work has started.
<message|>

<|message:impl-parser>
Prioritize parser tests first.
<message|>

You can send multiple messages in one turn. If the recipient specifies an invalid or unresolvable target, it is an error.

Messages support GFM-style markdown syntax. Messages also support inline links to local file content or URLs. You can also link directly to your workspace e.g. [foo]($M/foo.md).

When mentioning files in communications, always prefer using an inline link so that the file is unambiguous and accessible.

## Task and worker operations

Tasks and workers are managed with invoke calls using flat parameter blocks.

### Creating tasks

<|invoke:create-task>
<|parameter:id>impl-parser<parameter|>
<|parameter:title>Implement parser updates<parameter|>
<invoke|>

A task requires `id` and `title`. Use `parent` to nest under an existing task:

<|invoke:create-task>
<|parameter:id>impl-parser<parameter|>
<|parameter:title>Implement parser updates<parameter|>
<|parameter:parent>parser-feature<parameter|>
<invoke|>

### Spawning workers

<|invoke:spawn-worker>
<|parameter:id>impl-parser<parameter|>
<|parameter:message>Implement Unit A from the [plan]($M/plans/the-plan.md).<parameter|>
<invoke|>

`spawn-worker` creates a worker for a task. The `id` must match an existing task. The `message` parameter is the worker's initial instruction with the same semantics as a normal `<|message:TASK-ID>...<message|>`.

If a worker already exists on that task, `spawn-worker` kills the old worker and spawns a new one.

### Killing workers

<|invoke:kill-worker>
<|parameter:id>impl-parser<parameter|>
<invoke|>

Explicitly kills the worker on a task. Errors if no worker exists.

### Updating tasks

<|invoke:update-task>
<|parameter:id>impl-parser<parameter|>
<|parameter:status>completed<parameter|>
<invoke|>

Valid status values:
- `pending` — Reopens the task. Does not affect the worker.
- `completed` — Marks the task finished. Blocked if any child tasks are still pending or working.
- `cancelled` — Kills the worker (if present) and removes the task and its entire subtree from the task tree.

The `working` status is derived — it reflects that a task has a live worker. You never set it manually.

### Messaging workers

To send instructions or follow-ups to a worker, use `<|message:TASK-ID>`:

<|message:impl-parser>
Focus on test coverage before refactors.
<message|>
