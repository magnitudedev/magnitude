
# Grammar Leniency & Whitespace Design

This document describes the leniency rules and whitespace heuristics baked into the GBNF grammar generated by `grammar-builder.ts`.

---

## 1. Close Tag Leniency

The body DFAs accept **4 close tag variants** for every tag type (think, message, parameter, filter). This handles known model failure modes where the model deviates from the canonical form.

| Variant | Example | Mode |
|---------|---------|------|
| Canonical | `<parameter\|>` | — |
| Slash + pipe | `</parameter\|>` | Mode 1 |
| Slash, no pipe | `</parameter>` | Mode 2 |
| Bare (no slash, no pipe) | `<parameter>` | Mode 3 |

**Mode 1** (`</tag|>`) is the most commonly observed failure. Mode 2 follows if Mode 1 occurs. Mode 3 is low probability and not yet observed.

**Known tradeoff for Mode 3:** `<parameter>` matches the open tag syntax. If the model produces a bare open tag inside body content (e.g., writing about HTML), the DFA will interpret it as a close tag and terminate the body early. This is accepted as a rare edge case.

All accepted variants are normalized to the canonical form in history. The grammar accepts them as legitimate close tags.

### DFA structure

The body DFA is implemented in `generateBodyRules(prefix, tagName)`. For a tag named `parameter`:

```
body-s0:  [^<] → s0  |  "<" → s1
body-s1:  "/" → sl  |  "p" → s2  |  [^</p] → s0
body-sl:  "p" → s2  |  [^<p] → s0
body-s2:  "a" → s3  |  ...
...
body-s10: "|" → pp  |  ">" → tw0  |  "<" → s1  |  [^<|>] → s0
body-pp:  ">" → tw0  |  "<" → s1  |  [^<>] → s0
body-tw0: [ \t] → tw1  |  "\n"
body-tw1: [ \t] → tw2  |  "\n"
body-tw2: "\n"
```

The DFA has **no `""` exits** — it cannot terminate without consuming a close tag. This prevents premature body exit.

## 2. Invoke Close Leniency

The `invoke-end` rule accepts the same 4 variants for the `</invoke|>` close tag:

```gbnf
invoke-end ::= "<invoke|>" | "</invoke|>" | "</invoke>" | "<invoke>"
```

This is a flat alternation in the outer rule (not a DFA), since invoke close appears in a constrained position.

## 3. Whitespace Heuristics

### The Whitespace Trap

The core failure mode: `[ \t]*` before a mandatory character creates an **unbounded loop**. If the model's probability mass is on non-whitespace tokens (e.g., it wants to write content), the grammar constrains it to only produce spaces or tabs. The model gets stuck generating whitespace indefinitely because:

1. `[ \t]*` accepts zero or more spaces/tabs
2. The model cannot produce the mandatory character (e.g., `\n`) if its top tokens are content characters
3. Under constrained decoding, the model generates the only valid tokens — spaces and tabs — forever

This was the root cause of the original `hws "\n"` trap after parameter open tags.

### Design Rules

#### After open tags: mandatory `"\n"` for block elements, inline for parameters

```gbnf
lens  ::= "\n" hws "<|think:name>" "\n" think-body
msg   ::= "\n" hws "<|message:user>" "\n" msg-body
param ::= "<|parameter:name>" param-body
```

**Block elements** (lens, message, invoke): A single mandatory `"\n"` is required before body content starts. This:
- Prevents premature body closing
- Avoids the whitespace trap (no `[ \t]*` loop)
- The model almost always produces a newline after an open tag anyway

**Parameters are inline**: Parameters go directly into the body DFA with no newline requirement. The canonical format is `<|parameter:name>value<parameter|>` — open tag, value, close tag on the same line. Multi-line values are supported (the DFA accepts newlines in content), but the format does not force them.

**Why not `hws "\n"`?** The original grammar used `hws "\n"` (optional horizontal whitespace then newline) after open tags. This created the trap: the model wanted to write content but could only produce `[ \t]` or `\n`. If content tokens dominated, it would loop on spaces/tabs.

#### After close tags: bounded trailing whitespace in DFA (tw0/tw1/tw2)

The DFA includes trailing states after the close tag `>`:

```gbnf
tw0 ::= [ \t] tw1 | "\n"
tw1 ::= [ \t] tw2 | "\n"
tw2 ::= "\n"
```

This allows 0, 1, or 2 trailing spaces/tabs before a mandatory newline. The DFA does not terminate until the newline is consumed.

**Why bounded (tw0/tw1/tw2) instead of `[ \t]*`?** Unbounded `[ \t]*` before `"\n"` is the same trap. With `tw0/tw1/tw2`, each state is a fixed step — no cycle, no loop. The model can produce at most 2 trailing whitespace characters.

**Why is this safe after close tags?** After a close tag, the model wants to move on (next parameter, invoke close, etc.). Producing a newline is natural and high-probability. The model is not fighting the grammar here, unlike after an open tag where it wants to produce content.

#### Before open tags: `"\n" hws` (safe)

```gbnf
"\n" hws "<|think:..."
"\n" hws "<|invoke:..."
```

`"\n" hws` (newline then optional horizontal whitespace) is safe here because the model wants to produce the open tag `<`, which is always valid. The `hws` (`[ \t]*`) matches empty, so the model can go directly to `<`. No trap.

#### After yield: bounded trailing whitespace

```gbnf
yield ::= "\n" hws "<|yield:user|>" [ \t]? [ \t]? "\n"
```

Yield is the terminal element — nothing follows it in the grammar. Using `ws` (`[ \t\n]*`) after yield would create infinite generation (the model could loop on whitespace forever with no next element to force a transition). The bounded `[ \t]? [ \t]? "\n"` caps trailing whitespace at 2 characters.

#### Parameter count: bounded repetition

```gbnf
# For a tool with 3 parameters:
t-edit ::= ... (t-edit-p-path | t-edit-p-old | t-edit-p-new)? (t-edit-p-path | t-edit-p-old | t-edit-p-new)? (t-edit-p-path | t-edit-p-old | t-edit-p-new)?
```

Instead of `(param-a | param-b)*` (unbounded repetition), parameters are repeated `N` times with `?` (zero or one per slot). This caps total parameter count at N (one per parameter defined for the tool). The grammar cannot enforce uniqueness, but it prevents infinite parameter repetition.

**Why not `*`?** `(param-a | param-b)*` allows the model to repeat the same parameter indefinitely. With bounded `?` slots, the grammar terminates after at most N parameters.

## 4. Body DFA Design

### Shared DFAs

Four shared body DFAs are generated — one per close tag name:

| Rule | Close tag |
|------|-----------|
| `param-body` | `<parameter\|>` |
| `think-body` | `<think\|>` |
| `msg-body` | `<message\|>` |
| `filter-body` | `<filter\|>` |

All parameter rules share `param-body`. All lens rules share `think-body`. This is ~7x smaller than generating a separate DFA per parameter.

### DFA properties

- **No `""` exits**: The DFA cannot match empty string. Body content MUST end with a close tag.
- **Sole close tag handler**: Outer rules do not include literal close tags. The DFA is the only place close tags are consumed.
- **Trailing whitespace included**: The DFA consumes the close tag AND the trailing `[ \t]? [ \t]? "\n"`. Outer rules need no trailing whitespace handling.
- **Unambiguous `<` handling**: Any `<` that doesn't lead to a valid close tag is consumed as body content and the DFA returns to `s0`.

### Failure recovery

If the model produces a partial close tag that doesn't complete (e.g., `<param` then something unexpected), the DFA returns to `s0` and continues accumulating content. The partial match is treated as body content.
